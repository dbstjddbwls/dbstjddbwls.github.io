실습 - 게시글 자세히보기 제작

  • 기존 로그인과 회원가입, 게시글 작성 실습을 진행한 코드에 추가로 게시글 자세히 보기 기능을 추가해보았습니다.

  • 목표

    • Main 화면에서 보여지는 게시글 목록 중 하나를 클릭 시 해당 자세히보기 페이지로 넘어가며, 내용을 보여줌
    • 자세히 보기 페이지에서 이전 버튼 클릭 시 이전 화면으로 이동 (URL Typing으로 접속 할 경우를 대비)

  • 구현

    • 우선 기존 게시글 목록을 데이터로 받아와서 map으로 하나씩 화면에 뿌려주는 return을 Link 태그로 감싸줍니다.
      해당 태그의 to 속성에는 “/post/${postNum}”으로 지정합니다. (postNum의 경우 data내 포함된 게시글 번호)
    • 이후 Link 태그의 state 속성으로 post data를 전송합니다.
    • postDetail이란 페이지를 만들고 App.jsx에서 Routers > Route 태그를 하나 생성하여 path를 “/post/:postId”로 넣고 element를 로 넣어줍니다.
      이렇게하면 “/post/13”이란 경로로 들어가면 13이란 value를 가진 postId를 PostDetail 컴포넌트가 전달받고, 로 전달받은 state 값을 useLocation().state이란 Hook을 통해 확인할 수 있습니다.
    • 전달받은 state 내 postNum을 객체로 생성하여 axios 통신의 인자로 넣어 통신을 실행한 뒤, 정상 응답이면 전달받은 res의 data내 title이나 content, 조회수, 작성자 등을 useState에 넣어 화면에 보여줍니다.
    • useState를 쓰는 이유는 이후 수정 기능을 만들 때 setter 함수까지 보내줄 수 있게 구성하였습니다.
    • 해당 data를 화면에 출력해주면 페이지 자세히 보기가 완성되었습니다.
    • 이전 버튼의 경우 Button Tag의 onClick으로 (window).history.back()을 사용하였습니다.

  • 막힌 부분은?

    • axios 통신 시 계속 Code 400이 뜨길래 어떤 것이 잘못된 것인가 확인해보니 통신을 위해 줘야하는 req가 객체 타입이여야 하고 key와 value로 되어있어야 하는데, 해당 postNum만 보내 오류가 났었습니다.
    • 이전 버튼을 구현할 때 location.href(-1)로 가능한 줄 알고 구현해보았으나 실패하여 알아보니 .history를 쓰라고 되있었습니다.
    • Link Tag에서 state로 해당 post의 정보를 넘기면 되었는데, 이를 까먹고 자세히보기 페이지에서 모든 postList를 불러온 뒤 postId가 postNum과 동일한 경우를 하나하나 for문으로 찾으려 했습니다. 이후 state에 다시 검색하여 해당 불필요한 사태는 피했습니다.

Context API의 문제점

useContext로 watch 중인 Context가 update되면 컴포넌트가 re-render 되버린다.

flux

  • FaceBook에서 고안한 어플리케이션 아키텍쳐이다.
  • 단방향 데이터 흐름이란 특징을 가지고 있으며, Action을 Dispatcher로 보내고 해당 Action을 Store란 저장소에 보내고 화면에선 그 Store에 저장된 값을 가져온다.
  • 이후 Action의 값을 업데이트하면 해당 업데이트 값을 다시 Dispatcher로 보낸다
  • Action → Dispatcher → Store → View (→ Action Update → Dispatcher)

Redux

  • Reducer + Flux를 합친 단어
  • 모든 전역 State를 하나의 Object에 담아서 사용한다.

Action

  • action은 평범한 javascript의 객체이다.
  • type필드를 필수적으로 가지고 있어야 한다.
  • 일반적으론 payload 필드에 데이터를 넣는다.

순수함수란?

  • pureFunction으로 주로 사용하며, 동일한 인자가 같은 경우 항상 같은 값이 return 되도록 보장이 되야한다.
  • side-effect가 일어나면 안된다.
  • return 값으로만 소통하며 평가 시점이 중요하진 않다.

Reducer

  • Flux 아키텍쳐 중 dispatcher를 대체하는 부분이다.
  • 현재 state와 들어온 action을 이용하여 다음 state를 결정하는 순수함수이다.
  • state의 자료형은 자유이다.
  • 상태를 update하면 대입 연산이 아닌 다음 state를 반환한다.

Store

  • store에 reducer를 등록하여 쓸 수 있다.
  • dispatch된 action을 모든 reducer에 전달해준다
  • store가 처음 생성될 때 action을 reducer에게 전달하여 초기 state를 저장한다.
  • 실제 적용 코드
    • getState()를 통하여 counter

React-redux

  • react 컴포넌트 내에서 context를 통해 store에 접근 가능해짐
  • 본인이 소유하고 있는 state가 바뀌었을 때만 re-render
  • Contexdt API와 동일하게 Provider로 store 전달을 한다.
  • Store는 reference type이며 re-render가 되지 않아 useContext와는 다르다.

회고

오늘은 개인적인 실습과 Redux에 대하여 공부를 진행하였습니다. 실습의 경우 의무는 아니였지만 이후 CRUD 작업 시 게시판 기능을 만들 때 필수인 자세히보기 기능을 만들고 싶었고 해내었습니다.

만약 여기에 기능을 추가하면 이전이나 다음 글 보기, 수정하기 기능들을 추가할 수 있을 것 같지만, 당분간 다시 강의에 집중해야하여 나중에 시간이나면 추가로 기능 개발을 해봐야 할 것 같습니다.

redux의 경우 기존 온라인 강의로 배웠으나 생각보다 난이도가 있었고, 이번 실시간 강의로 좀 더 이해한 느낌입니다. 아직까진 store와 친숙하진 않지만 점점 실습에 사용해가며 전역 변수를 굳이 한 컴포넌트에서 자식 컴포넌트로 주는게 아닌 통괄적인 관리가 마음에 듭니다.
