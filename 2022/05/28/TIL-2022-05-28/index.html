수업 내용

Test

  • 특정 조건에서 시스템이나 구성 요소가 실행되고 결과가 관찰되고 기록하며 평가를 하는 활동
  • 테스트를 해야하는 이유
    • 이번 작업이 잘 작동하는지 검증
    • 기존 작업에 대한 side effect 검사
    • Reviewer이나 후임을 위한 가이드

unit Test

  • 일반적으로 함수나 컴포넌트 단위의 테스트를 작성
  • props 등 다른 컴포넌트 영향을 받는 경우 직접 주입해서 테스트
    = 함수의 인자를 직접 넣어서 호출하는 방식
  • 예) props로 특정 값을 넣었을 때 원하는 결과값이 나오는지

Integration Test

  • 단위 모듈이 결합되어있는 형태를 테스트

  • unit Test가 완료되었다면 해당 컴포넌트를 리턴하는 상위 컴포넌트에 props를 전달하여 컴포넌트끼리 정보를 받을 수 있는지 확인

  • 예)

jest

  • unit / Intergration Test 작성 및 실행, 결과 출력을 도와주는 테스팅 프레임 워크
  • CRA를 설치 시 자동 설치됨

expect

  • jest의 문법

  • expect 안에 무언가를 넣으면 다양한 메소드를 이용 가능하다. (jest, jest-dom)

  • CRA가 자동으로 babel 세팅을 해둬서, *.test.ts 파일에 대해 자동으로 넣어줌

React-Testing-Library

  • React 컴포넌트를 test할 수 있도록 renderer 등 함수 제공

  • CRA에 이미 설치되어 있다.

  • scripts에서 확인 가능하다

render, container

  • render는 react-testing-library의 함수
  • container는 render를 통해 생성된 최상위 부모 DOM
  • 일방적으로 js에서 다루는 HTML Element이므로 innerHTML같은 것도 활용 가능

Fire event

  • fireEvent는 react-testing-library의 함수

  • 특정 dom에 event 객체와 함께 event를 발생시킴

  • 아래와 같은 경우 idInput의 change event handler가 받는 event 객체는 {target: {value: ‘aaaaaa’}}가 된다.

e2e Test

  • End to End 테스트
  • 사용법은 실제 브라우저에 페이지를 렌더하고 사용자의 입장에서 어떤 결과가 나오는지를 검증
  • 예) 폼을 채우고 로그인 버튼을 누른 경우 환영합니다! url이 바뀌면서 환영합니다!를 렌더 테스트

+) async, await

  • 비동기화의 문법 중 하나
  • 기존 callback이나 promise then에서 벗어나기 위한 문법이다
  • async로 함수를 감싸서 선언하면 return값이 promise의 형태로 전달되게 된다.
  • 이렇게 감싸진 값을 await를 통하여 불러올 수 있다.
  • async는 promise를 return하기 때문에 뒤에 then을 붙혀서도 활용이 가능하다.
  • async의 값이 나와야 await가 실행 가능하기 때문에 동기적인 구조가 되는 것이다.

회고

오늘은 별도로 주말 특강을 신청하게 받아보았습니다.

현업에서도 실제로 자주 쓰는 Test에 관련한 내용이였고 기존 매번 값을 넣어가며 실행했어야 하였는데, 이번 수업에 배운 것으로 그럴 필요 없이 테스트 코드 작성 방법을 알게되었습니다.

이런 방식을 좀 더 활용하면 에러를 획기적으로 줄일 수 있을 것 같지만, 개발 시간이 넉넉해야 저는 가능할 것 같습니다.

추가적인 질문 시간에 async와 await에 대해 질문을 하여 짧게 설명을 들었는데 가끔 이러한 return 값을 동기적으로 써야할 때 써보면 좋을 것 같습니다.
