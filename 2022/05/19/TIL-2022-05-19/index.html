수업 내용

GraphQL

  • Graph Query Language의 약어로 서버로부터 데이터를 효율적으로 가져오기 위한 언어이다.
  • 참고 URL : https://www.howtographql.com/ (How to GraphQL)
  • 라이브러리가 아닌 하나의 명세이며, 다루기 위해선 구현체가 필요하다.
  • REST API vs GraphQL
    • REST API
      • 리소스의 모양과 크기는 서버에서 컨트롤
      • 다중 엔드포인트, URL과 Method에 따라 접근할 수 있는 데이터가 다르다.
      • 각각의 요청은 정확히 하나의 경로 처리 함수를 호출
      • 내가 사용자 이름만 알고싶어도 users/id를 호출하면 서버가 정해둔 모든 데이터를 다 받아오게 된다. (overfatching)
    • GraphQL
      • 클라이언트가 필요한 리소스를 따로 요청
      • 단일 엔드포인트로 GraphQL 스키마에 따라 요청하는 데이터의 구조에 따라 다르다.
      • 하나의 쿼리가 여러 resolver를 호출하여 여러 리소스가 포함된 중첩 응답을 구성한다.
      • 필요한 데이터만 요청할 수 있고 원라는 중첩 데이터를 요청할 수 있다.
      • 매번 서버에 API 요청을 하지 않아도 되고 백엔드에서도 분석이 가능하다.
      • 스키마 나 타입 시스템의 이점이 있다. (백엔드와의 구조를 동일시 할 수 있음)

  • REST API의 통신 과정. 원하는 정보를 얻기 위하여 요청을 분리한다.

  • GraphQL의 통신 과정. 요청시 정해진 구조를 보내서 정보를 한 번에 받는다.

  • 주요문법은 doc > example_queries 폴더 내에서 확인 가능하다.

GraphQL Learn

  • URL : https://graphql.org/learn/
  • GraphQL은 API를 위한 쿼리 언어이며 타입 시스템을 사용하는 서버사이드 런타임이다.
  • GraphQL은 타입과 필드를 정의하고 각 타입의 필드에 대한 함수로 구현된다.
    ⇒ 로그인한 사용자가 누구인지, 사용자의 이름이 무엇인지 타입을 정의하고 구조를 선언함
  • Alias (별칭) : 단 같은 이름의 구조는 두 번 이상 반복할 수 없다. 이럴땐 변수처럼 별칭을 붙혀줘야 한다.
  • Fragment (프래그먼트) : 반복되는 필드 셋 선언 시 React 처럼 하나로 묶어주는 역할을 한다.
    하나의 함수라고 생각하면 편하며 …을 통하여 값을 전달할 수 있다.
  • 작업 이름 : 작업 타입은 query, mutation, subscription 이 될 수 있으며 어떤 작업의 타입인지 기술한다. 기존까지 작업이 익명함수라면 이 기능은 함수명을 선언하는 것이라 생각하면 된다. 함수와 비슷한 만큼 변수를 전달할 수 있다.
  • Variables (변수) : 지금까지의 쿼리는 문자열로 전달이 되었다. alert로 쿼리문을 출력시 \n이 있는 것을 확인할 수 있는데 동적인 방법으로 쿼리를 조회할 필요가 있을 경우 변수를 사용하여 작업한다.
    GraphQL은 Query Variables란 기능을 제공한다.
  • 지시어 : if문과 같은 조건 기능을 사용할 수 있다.
    • @include(if: Boolean) : 인자가 true인 경우에만 해당 필드를 결과에 포함한다.
    • @skip(if: Boolean) : 인자가 true이면 해당 필드를 건너뛴다.

Apollo, Mutation

  • Mutation : Query가 데이터를 가져오는 작업이였다면 Mutation은 데이터를 수정하는 작업을 하는데 사용된다.
  • Mutation은 데이터를 바꾸기 때문에 제공하는 example을 찾아야한다.
  • 참고 URL : https://studio.apollographql.com/sandbox/explorer/
  • Apollo의 경우 다양한 기능이 추가된 graphQL의 라이브러리라고 보면 된다.
  • Query 필드는 병렬로 실행되지만 Mutation 필드는 하나씩 차례대로 실행된다. 값을 수정하는 영역이기 때문에 동시에 실행하면 문제가 생길 수 있어서이다.
  • Insert Mutation
    insert users()에 인자로 객체 형식의 값을 전달한다. 해당 값을 variables로 추가하여 작성한다.
  • Update Mutation
    update_users()에 인자로 where은 어떠한 값을 참고하여 변경할지와, 새롭게 세팅할 값을 받는다. 특정 id > _eq값을 찾아서 해당 요소의 name 값을 set의 값으로 변경해준다.

Scema & Type

  • 참고 URL : https://graphql.org/learn/schema
  • GraphQL은 스키마 및 타입 시스템으로 객체 타입과 필드를 받을 수 있다.
  • 객체 타입의 모든 필드는 0개 이상의 인자를 가질 수 있다.
  • 스키마의 대부분 타입은 일반 객체지만 스키마 내에는 위에서 본 Query와 Mutation이란 특수한 두 가지의 타입이 있다. 위 2가지 예제 또한 하나의 타입이다.
  • 객체 타입은 이름과 필드를 가지지만, 어떤 시점에서 이 필드는 데이터로 분석이 되야하는데 이것이 스칼라 타입이 필요한 이유이다.
    아래 코드에선 name과 appearIn은 스칼라 타입이다.
  • GraphQL은 기본 스칼라 타입이 제공된다.
    • Int : 부호가 있는 32비트의 정수
    • Float : 부호가 있는 소수의 값
    • String : UTF-8의 문자열
    • Boolean : true or false
    • ID : ID 스칼라 타입은 고유한 식별자를 나타낸다. ID 타입은 String과 같은 방식으로 보여지지만 ID로 정의하는 것은 사용자가 읽을 용으로 작성되지는 않는다.
  • 위 스칼라 타입 외에도 커스텀 스칼라 타입의 선언이 가능하다
  • 열거형 타입 : 스칼라 타입의 종류 중 하나로 enum이란 값을 사용하여 타입의 인자가 허용된 값인지 검증하는데 사용된다.
  • Non-Null : 유효성 검사에 사용되는 방법으로 !를 통하여 적용할 수 있다. 사용하게 되면 Null이 아닌 다른 값을 무조건 반환한다. 값이 없다면 빈 배열을 반환한다.
    • 예시 : [] or [’a’, null, ‘b’]
    • [String] : 배열과 String 모두 null일 수 있음
    • [String!] : 배열은 null일 수 있으나 String은 null일 수 없음
    • [String]! : 배열은 null일 수 없으며 String은 null일 수 있다.
    • [String!]! : 배열과 String 모두 null일 수 없다.
  • 인터페이스 : TypeScript와 동일한 기능으로 파입이 포함해야하는 특정 필드를 포함하는 추상적인 타입이다.
  • 입력 유형 : 변수로 값을 지저하여 Variables에서 입력을 받아 작동하는 형식입니다.

회고

오늘은 GraphQL에 대하여 공부를 하였습니다. 기존 REST API가 우세한 경향이 있었지만 최근 GraphQL이 조금 더 편리하고 사용하기 용이하며 사용하는 회사가 늘고 있다 들었습니다. 사용해보니 확실히 원하는 자료만 가져올 수 있는 점이 아주 매력적이였습니다.

기존 fetch의 경우 모든 데이터를 받아와야 했기에 불필요성이 가미된 데이터가 많이 있었지만 이로 인해 데이터를 받는데 더 빠른 속도와 편리함을 느낄 수 있었습니다.
