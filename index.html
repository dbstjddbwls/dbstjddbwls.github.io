React 공식 문서

  • Props : React가 사용자 정의 컴포넌트로 작성한 Element를 발견하면 해당 속성들을 해당 컴포넌트에 단일 객체로 전달하는데 이를 Props라고 한다. 또한 Props는 읽기 전용이며 props를 통하여 스스로 값을 변경하게 하는건 좋지 않다.

  • State

    • 기본적으로 this.state.comment 같이 this를 통해서 state를 갱신하면 렌더링되지 않는다. 꼭 setState를 사용하여 작업을 하자
    • state는 비동기적으로 업데이트 될 수 있다. 그래서 생명주기를 알아야한다.

  • 생명주기 : 특정 시점에 코드가 실행될 수 있도록 하는 기능이다. 즉 비동기성에 순서를 정해주는 것이다.

    • constructor() : 기본 값을 설정

    *state에 props를 복사하면 안된다. 버그를 일으키며 값이 변하더라도 state에 반영되지 않는다.

    • render() : render 함수 내 return을 화면에 출력한다.
    • componentDidMount() : 컴포넌트가 불려진 직후 호출된다. setState()를 호출하는 것도 가능하며 render()가 2번 호출되는 것이지만 브라우저가 화면을 갱신하기 전에 발생한다.
    • componentDidUpdate() : 최초 렌더링에는 호출되지 않으나 갱신이 된 직후에 호출된다. setState()를 사용할 순 있으나 그럼 setState()와 DidUpdate()가 무한반복되니 조건문을 활용하자.
    • componentWillUnmount() : 컴포넌트가 사라지기 직전에 호출된다. setState()를 호출해서는 안된다.

    회고

    실시간 수업에선 react의 실제 활용에 배운다면 온라인 강의는 개념의 설명과 이해에 기반한 것 같습니다. 특히 생명주기의 경우 알고는 있었지만 어떤 것이 어느 역할인지 몰랐는데 이번 강의로 비동기성에 순서를 줘서 원하는 대로 작업을 할 수 있지 않을까란 생각이 듭니다. 다만 실시간 강의가 타입 스크립트에 맞춰져 있는 만큼 온라인 강의도 타입스크립트를 기반으로 하고싶으나 강의 속도를 따라가며 타입스크립트 에러체크를 하려니 시간이 너무 많이 걸리는게 아쉽습니다.

토이 프로젝트(3)

토이프로젝트의 막이 내렸습니다.

아쉽습니다. 3일정도의 시간 내에 다 못 끝낼 양이란 것은 알았지만 원했던 일정을 그대로 소화하지 못한게 아쉽습니다.

짰던 일정은 총 금요일 오후, 월요일, 화요일 이 2.5일 동안 작업을 하는 것인데 여러 부분에서 막힘이 있었습니다. 어떤 것에 막혔었는지 기록하겠습니다.

  • JSON 파일의 날짜별 구분
    일단 토이프로젝트에는 오늘, 어제, 2일 전으로만 값이 나뉘어져 있었습니다. 단 저는 이것이 시간이 흐름에 따라 어제, 2일 전, 3일 전으로 바뀌는 등 시간을 변경하면 해당 값이 변경되게 진행하고 싶었습니다.
    일단 분기를 총 4가지를 나누었습니다. 기준은 현재 설정한 new Date(data)의 날짜로 부터 json 파일의 date를 parse합니다. ( ex. 2021-09-01의 값을 받으면 해당 값을 모두 split()하여 현재 new Date와 파트별로 비교한다) 이후 연, 월, 일, 오늘인지 확인을 합니다. 이후 각 해당되는 부분에 맞춰 작업을 진행하는 부분이 좀 늦었습니다. 추가로 정렬 방법에도 문제가 있었습니다. 다만 이에 대한 부분은 .reverse()를 통하여 완료하였습니다.
  • 많은 Obj
    Object의 객체를 만들어 안에 key:value의 값으로 많은 부분을 넣어야 했습니다. 계좌 이름부터 계좌 번호, 한도, 저금통, 사용 내역등을 뽑아내야했고 거기에 현재 사용금액과 한도에 비례하여 width값을 조정해야 했습니다.
  • 처음 써보는 Chart.js
    Chart.js를 알고는 있었지만 실제로 사용하니 생각보다 막히는 부분이 있었습니다. line type을 점선으로 만들거나 도넛 모양을 출력하거나 등 화면에 내가 원하는대로 출력하는 것이 조금 막힌 부분이 있었습니다. 다만 사용해보니 대용량의 데이터 출력에 맞는 부분인지는 조금 아쉬웠습니다. 아마 숙련도의 문제라고 생각되지만 이후 별도로 한 번 사용을 한 번씩 해보는 것이 재미있을 것 같습니다. 특히 line의 아래나 윗 부분을 칠하는 것이 멋있어 보였습니다.
  • 코드의 간결화
    저는 코드를 간결하게 쓰지 못했던 것 같습니다. 너무 재활용성을 생각하지 않고 그저 유지보수만 신경을 썼던 것 같습니다. 반복되는 코드만 수십 줄이 있었고 이것을 함수로 분명 줄일 수 있었습니다. 다만 그것을 빌드 계획 단계에서 생각을 못했습니다.

이번 토이프로젝트는 제가 기존 알던 것이 아닌 모르는 것들이 좀 더 많았습니다. 거기에 놓친 부분들도 있다보니 아쉬운 점이 많은 토이프로젝트가 아닌가 싶은 느낌입니다. 이후 미니 프로젝트나 또 다른 개별 작업을 한다면 이번 기회에 놓친 아쉬운 점들을 챙겨 다시 한 번 해보려합니다.

DOM, Element

  • DOM : 메모리의 웹 페이지 구조를 표현하는 브라우저가 이해하는 Element의 원형.
  • Element : 화면에 표시되는 모든 것.
  • JSX의 Function은 재사용 가능한 Element로 보며 Custom을 통하여 원하는 기능을 만들고 싶을 땐 Upper Case를 사용한다.

JSX, Babel

  • JSX : 문자도 HTML도 아닌 JS의 확장 문법. React.creatElement의 표현식
  • babel : JS의 컴파일러 종류 중 하나

JS와 JSX 섞어쓰기

위와 같이 function 내 if문을 써서 적용할 수도 있고, { }를 사용하여 js 문법을 사용할 수도 있다.

React의 Rerendering

  • 일반 JS에서는 변경이 실행되면 Element를 다시 그린다. 허나 React는 변경된 부분만 다시 그린다.
  • React의 Element는 불변객체이다.
  • 불변객체란 변하지 않는 객체란 뜻이며 우리는 그저 ReactDOM으로 전달만 하는것이고 변경과 판단, 반영은 리액트가 알아서 한다.
  • Element를 비교하는 경우
    • Element의 타입이 다른 경우, Element를 모두 버리고 다시 새롭게 구축한다.
    • Element의 타입이 같은 경우, 동일한 props의 내역은 유지하고 변경된 속성들만 갱신한다.
  • 해당 변경되는 요소를 버츄얼 DOM에 전달하고 변경 요소를 교체하고 비교한 뒤 결과를 화면에 보여준다.

이벤트 핸들러

  • 바닐라 JS의 경우 onclick, onhover, addEventListener … 등을 사용한다.
  • 리액트의 경우 비슷하게 사용될 수 있으나 onClick, onMouseOut 등 CamelCase를 사용한다.

훅 플로우 이해하기

  • hook flow란 hook들의 호출 타이밍을 말한다.
  • useState의 경우 prev란 기본 인자를 받는데 이전 값을 나타낼 수 있다.
  • useEffect()에서 [ ]를 아예 안주면 무엇을 하든 해당 Effect 함수가 실행된다.
  • useEffect()에서 [ ]를 빈 배열로 두면 맨 처음 렌더링 될 때만 실행되고 무엇을 해도 실행되지 않는다.
  • useEffect()에서 [ ]내에 원하는 useState key 값을 입력하면 해당 key값이 변경될 때마다 실행된다.

  • CompileOnSave : true / false(defual)로 구분하며, true로 설정하였을 때 컴파일을 어느 프로그램에서 진행하느냐를 정함
  • extends : 주로 상속받을 때 사용된다. 파일 경로명이 String Type으로 들어가야하며, ts 2.1 버전 이상부터 사용 가능한 속성이다. 특수한 설정 파일을 만들어서 tsconfig 파일을 수정없이 사용할 수 있다.
  • files, include, exclude : 이 3가지 속성 다 설정이 없으면 실행 폴더 내 모든 파일을 컴파일 한다. 어떤 파일이나 어떤 디렉토리를 컴파일 하는지 정하는 속성이다.

      • 상대 혹은 절대 경로의 리스트 배열이다
      • exclude보다 강한 영향력을 행사한다.
    • include, exclude
      • glob 패텁을 사용한다. (.gitignore 과 유사함)
      • include는 exclude보다 약하지만 *를 사용하면 .ts/ .tsx/ .td.ts 만 include 한다.

  • CompileOptions

      • TypeScript 2.0부터 사용 가능해진 내장 시스템이다.

      • 배열 안의 모듈이나 node_modules/@types 안의 모듈 이름에서 찾아온다.

      • lib를 지정하면 해당 lib 배열로만 라이브러리를 사용한다.

      • 컴파일 내용을 출력할 디렉토리

    • outFile

      • 단일파일로 출력 시 파일명

    • rootDir

      • 입력파일의 rootDir 설정

    • strict

      • 엄격한 타입 검사를 실행하는 옵션

      • —nolmplicitAny

        • 타입스크립트가 추론에 실패한 경우, any가 맞으면 any라고 지정하라함.
        • 아무것도 쓰지 않으면 에러
        • 해당 오류를 해결하면, any라고 지정되어 있지 않은 경우는 any가 아닌 것임

      • suppressImplicitAnyIndexErrors

        • noImplicitAny를 사용할 때 인덱스 객체에 인덱스 signature가 없는 경우 오류가 발생하는데 이를 예외 처리

      • —noImolicitThis

        • 명시적이지 않게 any 타입을 사용하여, this 표현식에 사용하면 에러가 발생함
        • 해결을 위해선 매개변수 인자 첫 자리에 this를 놓고, this에 대한 타입을 선언해야한다. class를 사용하여도 가능하다.
        • js에서 인자에 this를 넣으면 예약어라 에러가 난다.
        • call, apply, bind 등과 같이 this를 대체하여 함수 콜을 하는 용도로 쓰인다.
        • this는 보통 any로 선언한다.

      • —strictNullChecks

        • null 및 undefined 값을 모든 타입이 가질 수 없고, 가지려면 union type이나 any는 null과 undefined를 가진다.
        • 예외적으로 void는 undefined를 가진다.
        • 해당 옵션을 적용하지 않고 계속 사용하다보면 정확히 어떤 타입이 오는지 헷갈릴 수 있다.
        • null과 undefined를 사용하는 경우 신중하게 사용해야하며 되도록 정확한 타입을 지정하려는 노력을 해야한다.

      • —strictPropertyInitialization

        • 정의되지 않은 클래스의 속성이 생성자에서 초기화되었는지 확인한다.
        • 해당 옵션을 사용하려면 —strictNullChecks를 사용해야한다.
        • constructor를 사용 안하는 경우는 다른 함수로 이니셜라이즈 하는 경우엔 사용하지 않는다. (async)

      • —alwaysStrict

        • 각 소스 파일에 대해 js의 strict mode로 코드를 분석하고 엄격하게 사용을 해제한다. 즉 타입의 선언을 안해도 되는 것.

    • 브라우저 : HTML을 동적으로 만들기 위하여 브라우저에서 자바스크립트를 해석하고, DOM을 제어할 수 있도록 하는 자바스크립터 런타임 환경

    • Tuple
    • any
      • any는 어떤 타입이여도 상관이 없다.
      • 대신 이걸 최대한 쓰지 않는게 핵심이다.
      • 컴파일 타임에 타입 체크가 정상적으로 이뤄지지 않아 타입의 안전성을 잃는 것이 원인이지만, any를 써야하는데 쓰지 않으면 안되는 경우도 있다.
    • unknown
      • 응용 프로그램 작성 시 모르는 변수 타입을 묘사할 때 사용한다.
      • 해당 변수가 무엇이든 될 수 있음을 알려준다. any를 대신하여 나왔다.
      • any와 같이 아무것이나 할당할 수 있으나, 할당하려하면 컴파일러가 자동으로 추론한다.
      • 타입을 확정해주지 않으면 다른 곳에 할당도 안되고 사용도 안된다.
    • never
      • never Type은 모든 타입의 subtype이며, 모든 타입에 할당이 가능하다.
      • 하지만 never 자체에는 그 어떤 것도 할당할 수 없다.
      • any도 할당을 할 수 없으며, 잘못된 타입을 넣는 실수를 막을 때 사용한다.
    • void

타입 스크립트의 타입 시스템

  • 타입스크립트는 타입을 명시해야한다.
  • 타입을 명시하지 않으면 컴파일러가 자동으로 추론한다.
  • 즉 타입이란 해당 변수가 할 수 있는 일을 결정한다.

Structural Type System VS Nominal Type System

  • Structural type : 구조가 같으면 같은 타입이다.
  • Nominal type : 구조가 같아도 이름이 다르면, 다른 타입이다.

타입의 호환성

  • 같거나 서브 타입의 경우 할당이 가능하다. = 공변
  • 함수의 매개변수 타입만 같거나 슈퍼타입인 경우 할당이 가능하다. = 반병

타입 별칭

  • interface와 비슷하다
  • Primitive, Union Type, Tuple, Function
  • 직접 작성해야하는 타입을 다른 이름으로 지정할 수 있다.
  • 만들어진 타입의 refer로 사용하는 것이며 타입을 만드는 것은 아니다.
  • 즉 타입 제한을 변수로 만들어서 사용할 수 있다는 것이다.

회고

오늘은 처음으로 타입 스크립트에 대해서 배웠습니다. 이 부분에 대해선 항상 미지의 영역이였습니다. 다수의 기업의 채용 공고에 우대사항으로 늘 적혀있던 타입 스크립트를 배워보니 왜 우대사항에 두는지 알 수 있었습니다. 기존 자바스크립트의 추가적인 요소가 많았고 무엇보다 타입을 선언함으로써 보안적인 문제나 코드의 꼬임을 방지할 수 있는 것이 매력적으로 보였습니다. 강의가 두세번 밖에 없어서 아쉬울 정도였습니다. 이후 시간이 남는다면 타입스크립트만으로 하나정도 결과물을 만들어봐도 좋을 것 같습니다.

가져오기, 내보내기

  • 어떠한 파일의 기능을 내보내서 다른 파일에서도 사용할 수 있는 것을 모듈이라고 한다.
  • export default ~ : 해당 파일 내 어떠한 데이터를 내보낼 때 expor를 사용한다. default 형식이기 때문에 별도의 이름을 설정할 필요는 없고 import 하는 부분에서도 이름을 변경하여 사용할 수 있다. 단 기본 통로를 통해 보내는 것이기 때문에 하나의 데이터만 보낼 수 있다.
  • export name ~ : 위와 동일한 기능이지만 Name 형식이기 때문에 별도의 이름을 설정할 필요가 있다. 또한 import 하는 파일에서도 {}로 감싼 뒤, 동일한 이름을 사용해야하나, 이름을 import 파일에서도 A as B로 A란 이름을 B라고 바꿀 수는 있다. 단 이름을 설정하여 보내는 데이터기때문에 하나하나 지정하거나 *를 통하여 여러 데이터를 보낼 수 있다.

Lodash 사용해보기

  • _.uniqBy() : 데이터로 받은 배열 내 중복되는 값을 제거해준다.
  • _.unionBy() : 데이터로 받은 배열 내 중복되는 값을 제거해주는 건 동일하나, 여러 배열을 넣을 수 있다.
  • find() : 첫 번째 인자로 받은 배열 내에 두 번째로 받은 데이터를 찾는데 사용한다.
  • findIndex() : find와 기능 자체는 동일하나 찾은 데이터를 반환하는 것이 아닌 해당 데이터의 index를 반환한다.
  • remove(): 첫 번째로 받은 배열 내에 두 번째로 받은 데이터가 있다면 해당 배열에서 데이터를 삭제한다.

JSON

  • 데이터 타입 중 하나이며 Object 타입의 데이터이다.
  • 기본적으로 key와 value로 되어있으며 {key: value}로 작성한다.
  • 주로 비동기 브라우저나 서버 통신에 사용하며, 인터넷에서 자료를 주고 받을 때 주로 표현한다.
  • JSON.stringify() : 인자로 받은 data를 String 타입으로 하여 서술하여 보여질 수 있다. String 타입이 아니면 Object 타입이라 실제로 화면에 출력했을 때 오류가 날 수 있다.

Storage

  • Document나 window에 자체적으로 내장되있는 일종의 저장소이다.
  • 크게 local Storage와 session Storage가 있다.
  • local Storage : 페이지의 정보를 보관할 수 있다. 페이지를 닫아도 사라지지 않는다. 다만 사생활 보호 모드, 즉 시크릿 모드 중 생성된 local Storage는 해당 탭이 닫힐 때 사라진다.
  • session Storage : 페이지의 정보를 보관하는 기능은 동일하나 페이지를 닫으면 해당 정보는 사라진다.
  • 만약 한 번 코드에 넣고 실행한 뒤 setItem() 함수를 제거하여도 이미 localStorage에 정보가 들어갔기 때문에 제거되지 않는다.
  • 만약 정보를 수정하고싶다면 다른 변수를 만들어 정보를 넣고 setItem()으로 동일한 key를 설정한 뒤 변경될 값이 있는 데이터를 넣으면 된다.

Query String

  • String 타입의 데이터를 사용하여 페이지 내에서 검색이나 특정 값을 요청 때 사용한다.
  • host 주소?속성=값&속성2=값2&속성3=값3
  • host.com?key=value&s=name&date=2022

axios

  • api로 전달받은 정보를 js에서 활용할 수 있게 정리하여 데이터로 받을 수 있게 해주는 npm이다.

정규표현식

  • 정규표현식이란 문자열을 검색하고 대체, 추출하는데 사용 가능한 일종의 언어 패턴
  • 간단한 문자 검색부터 이메일, 패스워드 검사 등 복잡한 일치 기능을 빠르게 수행 가능하다.
  • 단 패턴이 수행 내용과 매칭이 되지 않아 가독성이 떨어지는 경우가 있어 어려워하는 경우가 많다.
  • 정규표현식 테스트 사이트
    • https://regex101.com/
    • https://regexr.com/ (★)
    • https://regexper.com/

정규표현식 생성 방법

  • 생성자 함수 방식 : new를 사용하여 생성한다.
  • 리터럴 방식 : /패턴/ 을 사용하여 생성한다. (주로 사용되는 방식)

정규표현식 메소드

  • RegExp.test() : 정규식 내 인자로 받은 문자열이 있는지 일치 여부를 반환한다.
  • String.match() : 문자열 내 인자로 받은 정규식을 확인하여 매칭되는 배열을 반환한다.
  • String.replace() : 문자열 내 인자로 받은 정규식을 확인하여 해당 문자열을 대체할 두 번째로 인자로 받은 데이터로 변환 후 반환한다.

플래그(Option)

  • g : 모든 문자가 일치해야한다.
  • i : 영어 대소문자를 구분하지 않고 일치한다.
  • m : 여러 줄이 일치해야한다.

패턴

  • ^ab : 줄 시작에 있는 ab와 일치해야 한다.

  • ab$ : 줄 끝에 있는 ab와 일치해야 한다.

  • . : 임의의 한 문자와 일치. 문자로 사용하고 싶다면 \를 사용해야한다.

  • a&verbar;b : a 또는 b와 일치해야 한다.

  • ab? : b가 없거나 b와 일치해야한다.

  • {n} : n개 연속 일치

  • {n, } : n개 이상 연속 일치

  • {n, m} : n개 이상 m개 이하 연속 일치

  • [abc] : a 또는 b 또는 c

  • [a-z] : a부터 z 사이의 문자 구간에 일치 (영어 소문자)

  • [A-Z] : A부터 Z 사이의 문자 구간에 일치 (영어 대문자)

  • [0-9] : 0부터 9 사이의 문자 구간에 일치

  • [가-힣] : 가부터 힣 사이의 문자 구간에 일치

  • \w : 63개 문자(대소영문자 52개, 숫자 10개, _)에 일치

  • \b : 63개 문자에 일치하지 않는 문자 (특수 문자)

  • \d : 숫자에 일치

  • \s : 공백에 일치 (탭, 스페이스바)

  • (?=) : 어떠한 값의 앞쪽이 일치해야 한다

  • (?<=) : 어떠한 값의 뒤쪽이 일치해야 한다.
