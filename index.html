수업 내용

기존 브라우저를 통해 페이지를 보는 과정

  1. 주소창에 URL을 입력
  2. 해당 주소로 HTTP get Request
  3. 해당 주소에 있는 서버거 HTTP Response로 HTML File을 보내준다.
  4. 브라우저가 해당 HTML 파일을 분석하여 화면에 보여준다.
  5. 새로운 페이지를 보여주고 싶다면 1~4를 반복

Single Page Application을 통한 과정

  1. 주소창에 URL을 입력
  2. 해당 주소로 HTTP get Request
  3. 해당 주소에 있는 서버거 HTTP Response로 HTML File을 보내준다.
  4. 브라우저가 해당 HTML 파일을 분석하여 화면에 보여준다.
  5. 새 페이지를 보여주고 싶으면 이동이 아닌 javascript를 이용하여 기존 DOM을 새 DOM으로 교체

SPA의 장단점

  • 장점 : 자연스러운 페이지 이동의 UX 제공
    (깜빡임 없음, 모바일처럼 animation 적용)
  • 단점 : 추가적인 공수(DOM 교체 작업) 필요

React-Router

  • React-Router란 React 프로젝트에서 SPA를 쉽게 도입할 수 있도록 도와주는 라이브러리

  • : 를 아래와 같이 동작하도록 만듦

  • - path : path내 경로와 현재 url이 일치하면 props로 보내는 element를 return한다. **가장 먼저 일치하는 Route 만 적용** - element : 반환할 React Element나 Component를 호출 - path 속성의 앞에 :를 붙이면 뒤 string을 key로 하는 값을 받을 수 있다.
  • : props.to 에 적힌 url로 http get request를 보내지 않고 브라우저에 표시되는 url만 변경 - to : a 태그의 href와 동일한 기능이다. 차이점은 anchor의 경우 브라우저 url을 이동하는 작업이라면, Link to=”url”의 경우 페이지를 이동하지 않고 URL만 변경하는 것이다. - to의 이동을 막고 싶다면 onClick 이벤트로 해당 event를 인자로 보내고 e.preventDefault()를 하면 된다.

회고

오늘은 온라인 강의로는 이전 배웠던 Next.js의 공식 문서를 읽는 강의를 하였고, 실시간 강의에서는 SPA의 특징과 React-Router를 통한 SPA 작업 방법에 대해 알아보았습니다.

흥미로웠던 점은 정말 새로고침 없이 깔끔하게 URL이 변경되며 화면이 바뀌는 점입니다. BrowerRouter이나 Routes, Route, path 등 신기한 점이 한 두가지가 아닙니다. 온라인 강의보다 확실히 실시간 강의가 실습과 함께 있어 그런가 오래 기억에 남는 느낌ㅇ입니다.

이후 SPA 특성을 사용하여 주로 실습을 하면 재밌게될 것 같습니다.

Next.js로 Blog 만들어보기

  • 참고 URL : https://nextjs.org/learn/basics/create-nextjs-app/setup

  • import Link Component

    • link를 넣는 컴포넌트로 a 태그와 같이 사용할 수 있다.

    • 두 태그의 차이점은 a 태그는 다른 페이지 항목을 처음부터 새로 불러들이는 것이라면, Link 태그는 기존 보고있는 페이지와 이동할 페이지를 비교하여 없는 파일이나 요소만 불러들여 화면에 출력한다. 즉 공통되는 파일이나 요소를 저장하는 이러한 점 때문에 페이지 이동 시의 깜빡임이 사라진다.
      이러한 것을 Code Splitting이라고 부른다.

    • 또 다른 기능으로는 prefetching이라는 기능이 있다.
      npm run build를 한 뒤 npm start를 하였을 때 보여지는 화면에 보여지는 정보만 가져오는 것이 아닌, 보여지는 화면에서 이동할 수 있는 링크의 페이지 정보까지 우선적으로 받아들인다. 이후 페이지 이동 시에는 chunk만 불러오게 된다. 이러한 기능 또한 a 태그만 사용해서는 불가능한 기능이다.

  • import Image Component

    • image 태그와 비슷한 기능을 하는 컴포넌트이다.

    • 속성을 동일하게 받으나 사용 사이즈에 맞춰 이미지 파일을 자동으로 최적화를 해주어 파일 크기도 줄어들게 된다.

  • MetaData

    • HTML에서 head 태그의 역할을 하는 기능이다.

    • 주로 란 태그를 써서 사용하며 안에 title 태그나 link favicon 태그 등을 사용하여 넣는다.

  • CSS Modules

    • React에서 CSS를 적용 시키는 방법 중 하나이다.

    • css 파일 자체로 적용 하는 것이 아닌, .module.css 파일로 만들어 적용하는 방법으로 특정 div나 컴포넌트의 className으로 module 파일 내 class 이름을 지정하면 자동으로 적용된다.

    • 해당 파일의 이름은 fileName.css가 아닌 fileName.module.css로 지정을 해야만 한다.

  • Global CSS

    • 모든 페이지에 전반적으로 들어갈 css 파일을 지정한다

    • 경로는 보통 처음 CNA 제작 시 자동으로 정해지며 pages 폴더 내 _app.js란 파일을 만들어 해당 파일 안에 Global CSS를 import 시킨다.

    • _app.js에 import된 css는 모든 파일에 공통적으로 자동 반영된다.

    • _app.js는 모든 컴포넌트를 한 번 감싸는 역할을 한다. 즉 css만이 아니라 js code도 공유가 된다.

    • SASS 또한 이용할 수 있다.

  • hydration
    페이지가 브라우저에 로딩되고 JS CODE가 실행되며 페이지가 인터렉팅하게 동작할 상태가 되는 과정이다.

  • pre-rendering

    • Next.js는 모든 페이지를 기본적으로 Pre Rendering을 하고 있다.
    • HTML 태그들을 먼저 우선적으로 pre-rendering을 한 뒤 이후 JS 파일들을 불러와서 적용시킨다. pre-rendering 이후 JS 파일을 적용시키는 과정을 위에서 말한 hydration이라고 부른다.
    • Next.js로 npm run start를 한 화면에서 js disabled를 하게되면 import나 alert 같은 코드들은 모두 제외되고 태그 형식의 컴포넌트나 이미지들만 보이게 된다.
    • 반면 create-react-app으로 만든 순수한 react-app에게 js disable은 치명적이다. React-app은 JS 실행으로 화면을 그리는 것이라 어떠한 화면도 나오지 않게 되기 때문이다.

  • getStaticProps()

    • 해당 함수를 통하여 마크다운이나 다른 폴더들을 읽어서 올 수 있다.

    • gray-matter라는 라이브러리로 파일의 해석을 진행해야 한다.

    • 따로 모듈화를 해두는 것이 좋으며 이러한 읽어본 파일들의 정보는 server 폴더 내에서 확인할 수 있다.

    • 이러한 기능은 build될 때 pre-rendering 된다.

    • 만약 사용자의 요청마다 외부의 데이터를 가져와야 한다면 getStaticProps가 아니라 getServerSideProps를 사용하면 된다.

  • Dynamic Routes

    • 파일을 읽고 불러들이고 만드는 역할을 하는 기능들이다.
    • getStaticPaths : [{ params : { id : string }}] 목록을 반환한다. params 안에는 dynamic routes key가 들어있어야한다.
    • getStaticProps: id 별로 md 파일에서 데티러를 읽어 page에 props로 전달하는 기능을 한다.
    • remark : npm으로 html에 md content를 보여주는 역할을 한다.
    • Catch-all Route : id를 하나만 받는 것이 아닌 여러 개를 받는 역할로 /pages/posts/[…id].js 이러한 방식으로 만들어두고 사용한다.
    • fallback : false면 404를 반환하고 true면 router.isFallback을 반환한다.

수업 내용

Apollo Server

  • GraphQL을 효율있게 사용할 수 있는 서버 모듈
  • 프론트와 서버 모두를 제공하는 라이브러리다.
  • express와 비슷하게 사용할 수 있으며 npm install apollo-server graphql을 받아 사용할 수 있다.
  • 이후 해당 폴더에 정의를 한 파일을 생성하고 안에 로직에 맞는 코드를 입력하면 사용 가능하다.
  • 실시간으로 값을 변경하고 싶다면 nodemon이란 모듈을 받아 package.json 내 script > start : nodemon filename.js라고 입력하면 지정한 파일을 불러와 쓸 수 있다.

Apollo Server Schema

  • Schema : 프론트쪽과 공유할 데이터의 구조를 뜻한다.
  • 실제 작업 시에는 한 폴더에 모든 설정 파일을 넣는게 아닌 분리해서 작업을 한다.
  • 주로 사용하는 폴더 경로
    • client
    • server
      • node_module
      • src
  • src 폴더 내 datasource 폴더나 index.js, schema.js, resolver.js 등을 넣어 사용함
  • REST API의 경우 apollo-datasource-rest라는 모듈을 받아서 사용하며 spaceX를 혼합하여 쓰는 경우가 많다.

Apollo Server Query Resolver

  • 쿼리 요청에 대한 응답을 생성하고 Data나 Promise를 반환한다.

  • Data는 보통 4가지 항목들을 전달받아 값을 반환한다.

    • parent : Resolver의 filed 부모에 접근해야할 때 전달한다.
    • args : 변수로 전달되는 항목을 요청한다.
    • context : 모든 Resolver가 실행될 때 공통적으로 반환받아야하는 부분들을 대상으로 한다.
    • info : Optional 적으로 필요한 정보를 요청한다.
    • 예시

  • 참고 URL : https://www.apollographql.com/tutorials/fullstack-quickstart/writing-query-resolvers

  • Paginate : Query에서 너무 많은 데이터를 가져오면 오류가 날 수 있다. 그걸 방지하기 위해 Page 형식 처럼 한 번에 보여지는 데이터의 갯수를 제한할 수 있다. 이로 인하여 서버의 부하를 줄인다.

  • Mutation Resolver : 주로 값이 수정되는 로그인이나 예약 등의 용도로 사용된다.
  • ApolloClient : 만약 프론트엔드와의 연결을 원한다면 해당 모듈을 사용하여 연결 후 화면에 출력한다.

GraphQL 요약

  • vs SQL : SQL은 DB랑 Server간의 요청이였다면, Server와 Web Client간의 요청이다.
    기존 Server에서 DB로 요청을 보내 해당 결과를 Client에게 보냈다면, GraphQL은 Client가 Server로 요청을 보낸 뒤 조건에 부합하는 응답을 준다.

  • vs REST API : 기존 REST API는 다중 엔드포인트로 곤혹이 있었으나, GraphQL은 단일 엔드포인트로 그것을 통하여 Overfetching이나 Underfetching 문제를 해결한다.

  • Overfetching : 기존 어떤 하나의 데이터를 뽑기 위해선 모든 데이터를 우선 받아와야 했지만 이러한 점을 개선하고 하나의 데이터만을 가져올 수 있다.

  • Underfetching : 기존 여러 요청을 하나하나 줘야했지만 GraphQL은 하나의 요청에 여러 조건을 담아 전달하여 한 번에 받을 수 있다.

  • GraphQL은 라이브러리가 아닌 언어 또는 명세이며 다루기 위해선 Apollo 같은 구현체가 필요하다.

  • graphiql express 등등 여러 도구가 존재한다.

  • Query : 요청을 보낼 때 사용하는 단어이며 원하는 결과를 받기 위해 데이터나 주석, 작업을 보낸다. 필드 객체를 참조하며 인자나 변수를 보낼 수 있다.

  • Fragment : 반복되는 필드셋이나 변수를 전달할 수 있다.

  • 변수, if문(지시어) : 동적으로 값을 전달하는 쿼리 방법이며 @include나 @skip을 통하여 사용한다.

  • Mutation : Query가 데이터를 받아오는 기법이라면 Mutation은 데이터를 수정하는 방법

  • Apollo graphQL : 다양한 기능이 추가된 라이브러리

  • Query와 다르게 Mutation은 순차적으로 실행된다. Query같이 데이터를 불러오는 것만 하면 순서가 상관 없지만 수정을 하기 위해선 이전 수정 내역에 기반하여 변경되는 점을 찾아야 할 수 있기에 순차적으로 실행된다.

  • inline Fragment : Fragment와 기능 자체는 동일하지만 interface나 union 타입일 때 주로 사용한다.

  • Type : 객체 타입과 필드를 가지는 시스템이며 타입 스크립트와 비슷하게 작동한다. Query, Scala, interface, union, input 등이 있다.

  • Query나 Mutation도 이러한 타입 중 하나에 포함되어있다.

  • Scala Type : String, float, Boolean 등 구체적인 데이터의 타입을 말한다.

  • Apollo GraphQL은 프론트와 서버를 모두 제공하는 라이브러리이다.

  • apollo-server graphql이란 모듈을 통하여 서버를 작동시킬 수 있다.

  • Schema란 프론트와 백엔드가 쉐어할 JSON 데이터의 구조 방식을 뜻한다.

  • Data Source의 경우 SpaceX REST API에는 기본적인 캐싱 기능이 포함되어 있다. 허나 Datasouce 자체에는 캐싱 기능이 없으니 추가 조치가 필요하다.

  • Resolver : 쿼리의 요청에 대한 응답 생성 코드를 뜻한다.

  • Pagenation : 서버 부하를 줄이는 기능으로 여러 데이터를 불러올 때 불러올 데이터의 갯수를 제한한다.

  • Mutation Resolver : 주로 로그인이나 예약 등의 정보가 변경될 기능에 사용되는 Resolver이다.

  • Apollo GraphQL로 프론트엔드 연결을 하기 위해선 ApolloClient란 모듈이 필요하다.

  • useQuery란 함수를 사용하여 data, loading, error 상황등에 대한 코드를 작성한다.

Next.js

  • Vercel이란 곳에서 만든 React의 프레임워크이다.
  • 라이브러리를 표방한 React의 장점은 살리며 다양한 편의기능을 추가한 프레임워크이다.
  • SSR(Server Side Rendering)를 기반으로 한다.
    • SSR이란 클라이언트로 전달될 HTML 파일 내용의 일부를 미리 그려서 내어주는 것을 뜻한다.
    • 클라이언트의 렌더링 속도를 빠르게 하여 사용자의 체감 속도를 증진시킨다.
    • 검색 엔진이 JS를 실행하지 않아도 크롤링이 가능하다.
  • 사용을 위해선 npx CRA을 기존에 사용했다면 Next.js는 npx create-next-app, 버전 관리 모듈인 n이라는 모듈의 설치가 필요하다.
  • Pre-rendering의 특징을 활용할 수 있다.
    • SSG : getStaticProps
    • SSR : getServerSideProps

useEffect()

  • Side Effect를 통해 DOM을 수정할 때 쓰는 Hook
  • 첫 번째 인자로 실행할 콜백 함수를 넘겨준다.
  • App 함수 컴포넌트가 다시 렌더링 되더라도 콜백함수는 첫 번째 렌더링 이후에만 실행한다. 즉 셋팅할 땐 실행하지 않으며 셋팅이 다 된 이후 실행한다.
  • 두 번째 인자로는 Array를 받으며 최초 렌더링 이후 1회 실행한다. 이후 해당 Array에 넣은 값이 바뀌면 첫 번째 인자로 받은 함수를 재실행 된다.
  • 만약 useEffect가 특정 value값 변경에 의해 Event가 쌓이게 된다면 useEffect 첫 인자의 함수 내 return 값으로 removeEvent를 해주면 된다. (Cleanup)

useMemo()

  • return으로 반환하는 값이 있는 연산이나 코드의 경우 계속 반복하는 것이 아닌, 기존 값을 저장해놨다가 인자가 변화하지 않으면 저장값을 반환하는 Hook
  • 첫 번째 인자로는 실행 함수를 넣는다. 해당 함수는 첫 렌더링 때 한 번 실행되고, 두 번째 인자로 받는 State나 Value가 변화하지 않는 이상 중복 실행이 된다면 이전 렌더링 때 실행되어 반환된 값을 전달한다.
  • 예시)
    input의 입력에 따라 state가 변하는 컴포넌트를 다른 컴포넌트에 해당 state를 props를 전달하고, 해당 자식 컴포넌트 안에 1~4억까지 모든 수를 더한 값을 하는 연산이 있다면 첫 렌더링 때 수행 후 부모 컴포넌트가 바뀔 때만 수행하고 자식 컴포넌트가 다시 렌더링 될 땐 실행하지 않고 이전 렌더링 값을 저장해뒀다 반환한다.

useCallback()

  • useMemo의 경우 첫 인자로 받은 실행 함수 내 return 값이 반환된다.
  • useCallback()의 경우 첫 인자로 받은 실행 함수 자체라 Memoization 된다.
    • 함수를 반환해야할 때 : Memo, Callback
    • 그 외나 변수를 반환할 때 : Memo
  • useCallback의 경우 첫 번째 인자 자체를 반환하는 것이기 때문에 cleanup이 불가능하다.

회고

오늘은 GraphQL의 온라인 강의 마무리와 Next.js의 시작을 진행하였습니다. GraphQL은 처음 시작할 땐 알기 쉬웠으나 심화로 들어갈 수록 너무 어려웠고 마지막은 거의 이해하지 못한 것 같습니다. 추후 리액트 강의 때 GraphQL에 관한 실시간 강의가 있던 것으로 기억하는데 이후 해당 수업을 듣고 추가로 공부를 더 해야할 것 같습니다.

Next.js의 경우 아직 환경 설정 및 파일 생성만 진행하여 이것이 CRA와 무슨 차이점이 있는지 아직 느끼진 못하였으나 실제 작업에서 용이하게 쓰이는 것으로 알고 있어 집중해서 공부해야 할 것 같습니다.

실시간 강의는 저번 useState에 이어 useEffect와 useMemo, useCallback 등 대표적인 hook에 대하여 배웠습니다. 이 중 useEffect의 경우 DOM을 제어하는 만큼 잘 배워둬야 이후 React 강의나 실습 때 쓸 수 있으리라 생각합니다.

GraphQL

  • Graph Query Language의 약어로 서버로부터 데이터를 효율적으로 가져오기 위한 언어이다.
  • 참고 URL : https://www.howtographql.com/ (How to GraphQL)
  • 라이브러리가 아닌 하나의 명세이며, 다루기 위해선 구현체가 필요하다.
  • REST API vs GraphQL
    • REST API
      • 리소스의 모양과 크기는 서버에서 컨트롤
      • 다중 엔드포인트, URL과 Method에 따라 접근할 수 있는 데이터가 다르다.
      • 각각의 요청은 정확히 하나의 경로 처리 함수를 호출
      • 내가 사용자 이름만 알고싶어도 users/id를 호출하면 서버가 정해둔 모든 데이터를 다 받아오게 된다. (overfatching)
    • GraphQL
      • 클라이언트가 필요한 리소스를 따로 요청
      • 단일 엔드포인트로 GraphQL 스키마에 따라 요청하는 데이터의 구조에 따라 다르다.
      • 하나의 쿼리가 여러 resolver를 호출하여 여러 리소스가 포함된 중첩 응답을 구성한다.
      • 필요한 데이터만 요청할 수 있고 원라는 중첩 데이터를 요청할 수 있다.
      • 매번 서버에 API 요청을 하지 않아도 되고 백엔드에서도 분석이 가능하다.
      • 스키마 나 타입 시스템의 이점이 있다. (백엔드와의 구조를 동일시 할 수 있음)

  • REST API의 통신 과정. 원하는 정보를 얻기 위하여 요청을 분리한다.

  • GraphQL의 통신 과정. 요청시 정해진 구조를 보내서 정보를 한 번에 받는다.

GraphQL 사용법

  • 공식 Github : https://github.com/graphql/swapi-graphql
  • 주요문법은 doc > example_queries 폴더 내에서 확인 가능하다.

GraphQL Learn

  • URL : https://graphql.org/learn/
  • GraphQL은 API를 위한 쿼리 언어이며 타입 시스템을 사용하는 서버사이드 런타임이다.
  • GraphQL은 타입과 필드를 정의하고 각 타입의 필드에 대한 함수로 구현된다.
    ⇒ 로그인한 사용자가 누구인지, 사용자의 이름이 무엇인지 타입을 정의하고 구조를 선언함
  • Alias (별칭) : 단 같은 이름의 구조는 두 번 이상 반복할 수 없다. 이럴땐 변수처럼 별칭을 붙혀줘야 한다.
  • Fragment (프래그먼트) : 반복되는 필드 셋 선언 시 React 처럼 하나로 묶어주는 역할을 한다.
    하나의 함수라고 생각하면 편하며 …을 통하여 값을 전달할 수 있다.
  • 작업 이름 : 작업 타입은 query, mutation, subscription 이 될 수 있으며 어떤 작업의 타입인지 기술한다. 기존까지 작업이 익명함수라면 이 기능은 함수명을 선언하는 것이라 생각하면 된다. 함수와 비슷한 만큼 변수를 전달할 수 있다.
  • Variables (변수) : 지금까지의 쿼리는 문자열로 전달이 되었다. alert로 쿼리문을 출력시 \n이 있는 것을 확인할 수 있는데 동적인 방법으로 쿼리를 조회할 필요가 있을 경우 변수를 사용하여 작업한다.
    GraphQL은 Query Variables란 기능을 제공한다.
  • 지시어 : if문과 같은 조건 기능을 사용할 수 있다.
    • @include(if: Boolean) : 인자가 true인 경우에만 해당 필드를 결과에 포함한다.
    • @skip(if: Boolean) : 인자가 true이면 해당 필드를 건너뛴다.

Apollo, Mutation

  • Mutation : Query가 데이터를 가져오는 작업이였다면 Mutation은 데이터를 수정하는 작업을 하는데 사용된다.
  • Mutation은 데이터를 바꾸기 때문에 제공하는 example을 찾아야한다.
  • 참고 URL : https://studio.apollographql.com/sandbox/explorer/
  • Apollo의 경우 다양한 기능이 추가된 graphQL의 라이브러리라고 보면 된다.
  • Query 필드는 병렬로 실행되지만 Mutation 필드는 하나씩 차례대로 실행된다. 값을 수정하는 영역이기 때문에 동시에 실행하면 문제가 생길 수 있어서이다.
  • Insert Mutation
    insert users()에 인자로 객체 형식의 값을 전달한다. 해당 값을 variables로 추가하여 작성한다.
  • Update Mutation
    update_users()에 인자로 where은 어떠한 값을 참고하여 변경할지와, 새롭게 세팅할 값을 받는다. 특정 id > _eq값을 찾아서 해당 요소의 name 값을 set의 값으로 변경해준다.

Scema & Type

  • 참고 URL : https://graphql.org/learn/schema
  • GraphQL은 스키마 및 타입 시스템으로 객체 타입과 필드를 받을 수 있다.
  • 객체 타입의 모든 필드는 0개 이상의 인자를 가질 수 있다.
  • 스키마의 대부분 타입은 일반 객체지만 스키마 내에는 위에서 본 Query와 Mutation이란 특수한 두 가지의 타입이 있다. 위 2가지 예제 또한 하나의 타입이다.
  • 객체 타입은 이름과 필드를 가지지만, 어떤 시점에서 이 필드는 데이터로 분석이 되야하는데 이것이 스칼라 타입이 필요한 이유이다.
    아래 코드에선 name과 appearIn은 스칼라 타입이다.
  • GraphQL은 기본 스칼라 타입이 제공된다.
    • Int : 부호가 있는 32비트의 정수
    • Float : 부호가 있는 소수의 값
    • String : UTF-8의 문자열
    • Boolean : true or false
    • ID : ID 스칼라 타입은 고유한 식별자를 나타낸다. ID 타입은 String과 같은 방식으로 보여지지만 ID로 정의하는 것은 사용자가 읽을 용으로 작성되지는 않는다.
  • 위 스칼라 타입 외에도 커스텀 스칼라 타입의 선언이 가능하다
  • 열거형 타입 : 스칼라 타입의 종류 중 하나로 enum이란 값을 사용하여 타입의 인자가 허용된 값인지 검증하는데 사용된다.
  • Non-Null : 유효성 검사에 사용되는 방법으로 !를 통하여 적용할 수 있다. 사용하게 되면 Null이 아닌 다른 값을 무조건 반환한다. 값이 없다면 빈 배열을 반환한다.
    • 예시 : [] or [’a’, null, ‘b’]
    • [String] : 배열과 String 모두 null일 수 있음
    • [String!] : 배열은 null일 수 있으나 String은 null일 수 없음
    • [String]! : 배열은 null일 수 없으며 String은 null일 수 있다.
    • [String!]! : 배열과 String 모두 null일 수 없다.
  • 인터페이스 : TypeScript와 동일한 기능으로 파입이 포함해야하는 특정 필드를 포함하는 추상적인 타입이다.
  • 입력 유형 : 변수로 값을 지저하여 Variables에서 입력을 받아 작동하는 형식입니다.

Redux

  • 컴포넌트 사이의 전역 상태간 상태관리 시스템이다.
  • 사용법은 npx create-react-app my-app —template redux를 작성하여 사용
  • typescript를 사용하고 싶다면 redux부분에 redux-typescript를 덧붙힌다.
  • 이후 npm install react-redux를 실행시키거나 npm install @types/react-redux를 입력한다.

  • useDispatch() : useDispatch()란 함수 안에 넣으면 dispatch 함수를 만들 수 있다. 액션 객체나 속성값 자체를 전달하며 작동시킨다.

  • store : 스토어는 사태가 관리되는 공간이다. 컴포넌트와 스토어는 별개의 공간이 있어, 스토어 안에 앱에 필요한 정보나 상태를 저장한 뒤 이후 필요할 때 꺼내서 사용한다.

data-fns

  • moment나 dayjs 처럼 일자를 관리하는 라이브러리
  • 날짜 관리 라이브러리 중 가장 많이 사용된다.
  • npm install date-fns date-fns-tz 를 통해 사용한다
  • dayjs나 moment랑은 다른 문법을 사용한다.

Styled-Component

  • CSS in JS를 이용한 작업을 뜻한다.
  • 글로벌 변수를 지양해야하는 JS와 대치되거나 css 파일 로드 타이밍 이슈가 있거나 쓰지 않는 css인지 구별이 안되는 등의 문제가 css에 있어 나타난 기능 중 하나이다.
  • JS의 경우 사용하지 않는 변수의 경우 표시를 해주기 때문에 알아보기 쉽고 확장도 용이하다.

Emmotion

  • 일반 styled는 백틱을 이용해야 하지만 emmotion의 경우 객체를 전달할 수도 있다.
  • css props를 사용하는데 jsx를 대체할 수 있다.
  • 부모 요소 조건부 선택자나 Media Query도 사용 가능하다.
  • 사이즈나 속도, 트렌드가 전반적으로 emmotion을 사용하는 편이다.

SASS

  • react에서도 SASS 이용이 가능하다
  • 따로 scss를 npm install 하지 않아도 사용 가능하다.
  • 변수를 선언하고 @extend 한 뒤 추가 작성하여 확장하는 등의 기능도 동일하게 사용 가능한다.
  • if문도 사용이 가능하다

회고

오늘은 주로 css에 관한 수업이 진행된 것 같습니다. 날짜를 지정하는 dayjs나 date-fns, moment와 함께 css에 사용되는 styled부터 emmotion 등 까지 궁금했던 부분들에 대해 다뤘던 점이 좋았습니다. 특히 scss를 자주 사용하고 싶은 입장에선 매우 좋은 강의였던 것 같습니다.

수업 내용

Context

  • Context란 컴포넌트 트리를 넘어 데이터를 공유할 수 있는 방법
  • 일일히 props를 넘겨지주 않고도 트리 전체에 데이터를 제공할 수 있다.
  • 주로 유저가 로그인을 했는지나 테마, 선호 언어 등에 이용된다.
  • 여러 레벨에 props를 넘기는걸 대체한다면 context보다 ㅋ컴포넌트 합성이 나을 수 있다.

Portal

  • DOM 계층 구조 바깥에 있는 DOM 노드로 자식을 렌더링 하는 방법
  • CreatePortal은 부모 컴포넌트 DOM 트리로부터 벗어나는 기능을 제공한다.
  • 이벤트의 경우 Portal이란 다른 컴포넌트에 있더라도 트리를 통해 전파된다.

Render Props

  • 코드의 재사용 방법 중 하나이다
  • React 컴포넌트 간에 코드를 공유하기 위해 함수 props를 이용하는 것
  • 즉 무엇을 렌더링할 지 알려주는 함수이다.
  • PureComponent가 이에 최적화 되어있다. props, state를 얕은 비교를 하여 성능을 최적화한다.

PropTypes

  • Props의 타입을 확인하기 위한 도구
  • link, Flow, TypeScript 같은 정적인 타이핑 도구이다.
  • 개발 모드에서만 동작한다. 유효하지 않은 prop type에 대한 경고를 준다.
  • 정규식같은 것으로 if문을 쓰는 등 커스텀이 가능하다

회고

수업 내용

React Event

  • React의 경우 SyntheticEvent 객체를 전달 받는다.
  • 예를 들어 onMouseLeave에서 event.nativeEvent는 mouseout 이벤트를 가리킨다. 즉 js와는 조금 다른 부분이 있다.
  • React는 소문자 대신에 카멜 케이스를 사용한다.
  • jsx를 사용하여 문자열이 아닌 함수로 이벤트 핸들러를 전달한다.
  • HTML과 다르게 return false를 반환해도 기본 동작을 방지할 순 없다. 반드시 .preventDefault를 사용해야 한다.
  • React의 경우addEventListener를 호출할 필요가 없다. 대신 처음 렌더링 될 때 리스너를 제공하면 된다.
  • 이벤트 버블링 : 어떤 컴포넌트나 엘리먼트를 클릭했을 때 부모 요소로 이벤트를 올려보내는 것
  • 이벤트 캡쳐링 : 자식 요소 중 어떤 요소가 이벤트를 발생시켰는가 확인하는 요소이다.
  • 순서는 보통 캡쳐링 이후에 버블링이 발생한다.

조건부 렌더링

  • React의 조건부 렌더링은 JS의 조건 처리와 같이 동작한다. if나 논리 연산자 사용도 가능하며 추가로 컴포넌트의 출현도 조절할 수 있다.
    • if : JS와 동일하게 작성
    • && : JS와 조금 다른데 앞의 조건이 만족하면 바로 뒤의 코드를 실행
    • 삼항 연산자 : JS와 동일하게 작성
    • 무시해야할 경우 : return null

List & Key

  • React에선 배열을 기반으로 엘리먼트 리스트나 컴포넌트 리스트로 만들 수 있다.
  • .map()을 사용하여 여러 태그를 만들 때 해당 html tag에는 꼭 key값을 넣어줘야한다. 안그러면 경고가 표시된다. 오류가 나진 않는다.
  • 일반 컴포넌트 선언이 아닌 map 함수 안의 요소에 써야한다.

Hook

  • Hook이란 useState를 기본으로 사용한다.

  • Class의 단점을 보완하며 재사용성이 강화되었다.

  • useState는 인자로 초기 state 값 하나를 받는다.

  • Hook은 2가지 규칙을 준수해야한다.

    • 최상위에서만 Hook을 호출해야한다. (반복문, 조건문, 중첩 함수 내에서 Hook을 실행해선 안된다.)
    • React 함수 컴포넌트에서만 Hook을 호출해야한다. js 함수 내에서는 안된다.

  • useEffect는 side effects에 사용된다. 주로 렌더링 이후에 어떤 작업을 할지를 담당한다.

  • componentDidMount & componentWillUnmount를 useEffect 하나에서 처리한다.

  • 사용 후 clean-up(정리)하는 과정이 필요하다.

  • 데이터 fetch나 구독, Dom 수정 등을 할 수 있다.

React 공식 문서

  • Props : React가 사용자 정의 컴포넌트로 작성한 Element를 발견하면 해당 속성들을 해당 컴포넌트에 단일 객체로 전달하는데 이를 Props라고 한다. 또한 Props는 읽기 전용이며 props를 통하여 스스로 값을 변경하게 하는건 좋지 않다.

  • State

    • 기본적으로 this.state.comment 같이 this를 통해서 state를 갱신하면 렌더링되지 않는다. 꼭 setState를 사용하여 작업을 하자
    • state는 비동기적으로 업데이트 될 수 있다. 그래서 생명주기를 알아야한다.

  • 생명주기 : 특정 시점에 코드가 실행될 수 있도록 하는 기능이다. 즉 비동기성에 순서를 정해주는 것이다.

    • constructor() : 기본 값을 설정

    *state에 props를 복사하면 안된다. 버그를 일으키며 값이 변하더라도 state에 반영되지 않는다.

    • render() : render 함수 내 return을 화면에 출력한다.
    • componentDidMount() : 컴포넌트가 불려진 직후 호출된다. setState()를 호출하는 것도 가능하며 render()가 2번 호출되는 것이지만 브라우저가 화면을 갱신하기 전에 발생한다.
    • componentDidUpdate() : 최초 렌더링에는 호출되지 않으나 갱신이 된 직후에 호출된다. setState()를 사용할 순 있으나 그럼 setState()와 DidUpdate()가 무한반복되니 조건문을 활용하자.
    • componentWillUnmount() : 컴포넌트가 사라지기 직전에 호출된다. setState()를 호출해서는 안된다.

    회고

    실시간 수업에선 react의 실제 활용에 배운다면 온라인 강의는 개념의 설명과 이해에 기반한 것 같습니다. 특히 생명주기의 경우 알고는 있었지만 어떤 것이 어느 역할인지 몰랐는데 이번 강의로 비동기성에 순서를 줘서 원하는 대로 작업을 할 수 있지 않을까란 생각이 듭니다. 다만 실시간 강의가 타입 스크립트에 맞춰져 있는 만큼 온라인 강의도 타입스크립트를 기반으로 하고싶으나 강의 속도를 따라가며 타입스크립트 에러체크를 하려니 시간이 너무 많이 걸리는게 아쉽습니다.
