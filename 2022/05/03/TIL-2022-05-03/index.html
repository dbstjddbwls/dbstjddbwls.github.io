Keyof & type lookup

Bundler

  • 번들러란 모든 종류의 파일들을 모듈 단위로 나누어 최소한의 파일 묶음(번들)로 만들어낸 것이다.
  • node_module 들을 생각하면 편하다.
  • Parcel : 구성이 없는 자동 번들링. 소, 중형 프로젝트에 적합하다.
  • Webpack : Parcel에 비해 매우 꼼꼼한 구성이며 중, 대형 프로젝트에 적합하다.

Parcel

  • 설치 방법 : npm i -D parcel-bundler

  • postcss : 일상적인 css 동작을 자동화하기 위해 자바스크립트 기반 플러그인을 사용하는 소프트웨어 개발 도구이다. 현재 autoprefixer 10.2.5 버전은 postcss 8.2.8 버전과 충돌이 난다. 이 경우 autoprefixer의 버전을 9로 낮추면 된다.

  • autoprefixer : 공급업체접두사를 자동으로 확인해주는 기능이다. webkit-box나 ms-등 최신 기술을 우선 적용해보고 안되면 취소하게 된다. browserslist란 package.json 내 속성을 사용하며, npm 프로젝트에서 지원할 브라우저 범위를 명시한다.

  • babel : ES 2015+ 이상의 문법 코드를 그 이전 JS엔진에서 실행할 수 있게 변환을 해준다. 설치는 @babel/core @babel/preset-env 등을 사용하여 추가한다. 추가로 위에 사용한 browserslist에 값을 설정했다면 그 범위에 맞게 작동한다.

  • CLI : 커맨드 라인 인터페이스라고 하며, npm i나 git 등 간단한 명령어를 통하여 작업하는 것을 통칭한다.

    ex) parcel index.html, parcel build index.html, parcel watch …

  • 배포

    1. .gitignore을 만들어 안에 .cache, .DS_Store, node_modules, dist를 제외하고 배포한다.
    2. git init → git status → git add . → git status → git commit -m “msg” → git remote add origin “github link” →git push origin master

Webpack

  • 설치 방법 : npm i -D webpack + webpack-cli webpack-dev-server@next

  • 적용 방법

  • entry, output

    • entry : 파일을 읽어들이기 시작하는 진입점 설정
    • output : 결과물을 반환하는 설정

  • plugin : 번들링 후 결과물의 처리 방식 등 다양한 플러인 설정

  • module화

  • autoprefixer(postcss)

  • babel

  • Netlify 배포

    1. .gitignore을 만들어 안에 .cache, .DS_Store, node_modules, dist를 제외하고 배포한다.
    2. git init → git status → git add . → git status → git commit -m “msg” → git remote add origin “github link” →git push origin master
    3. app.netlify.com 접속
    4. New Create site Click
    5. Github 내 해당 저장소 검색 후 선택
    6. 사이트 배포 전 옵션 선택 후 Deploy Site Click

NPX, Degit

  • npx degit ~ 명령어를 통해 github 저장소를 로컬에 다운 받을 때 사용한다.
  • 사용법 : npx degit UserName/RepositorieName newLocalFolderName
  • 터미널에서 버전관리가 되어있지 않을 때 git init을 통해 새롭게 버전관리를 할 수 있다는 장점이 있다.
  • git clone githubLink를 통하여 작업하면 버전의 내역까지 넘어온다. 즉 템플릿으로써는 부적합하다.

회고

오늘은 TypeScript의 마지막 강의와 Bundler에 대해 공부하였습니다. TypeScript의 경우 하면 할 수록 매우 편리하고 왜 여러 기업에서 TypeScript를 사용하는지 알 수 있었습니다. 매우 안정적이고 오류를 줄여주고 협업 시에도 유리하다고 판단이 되며 해당 언어에 대해 좀 더 깊게 공부해봐도 좋을 것 같다고 생각이 듭니다.

Bundler의 경우 기존 Parcel을 통하여 공부하였지만 Parcel이 무엇이 부족하고 어느 경우에 써야하는지 제한되는 부분을 알게 되었고 추후 중견 규모의 프로젝트를 들어가게 되면 주로 Webpack을 메인으로 쓸 것 같다는 생각이 듭니다.

당장 한 번만으로 TypeScript와 Webpack을 모두 알 수 있다고 생각하지 않습니다. 앞으로의 js 과제의 경우 모두 TypeScript로 대체하여 작업을 해보도록 노력하겠습니다.
