프로그램 언어의 역사

기존 2진법이나 16진법같은 머신코드로 프로그램을 짜기 어려워서 **어셈블러**란 것을 만들게됨

어셈블러

  • 머신 코드를 오브젝트 코드로 해석해주는 번역 프로그램
  • 기존 엄청 긴 머신코드로 더하기란 기능을 쳐야했으면 a=1; 만 쳐도 머신코드를 컴파일하여 메모리에 입력하게됨
  • 진행 방식
    Code → Token → AST → 머신코드
  • 해당 방식을 쓰는 언어 : C, JAVA …

AST explorer Link : https://astexplorer.net/

HTML, HTTP, Browser

어셈블리 이후 나타난 것이 **HTML, HTTP, Browser** 이다.

  • HTML = 쉽게 코드를 작성할 수 있고
  • HTTP = 쉽게 요청을 보내거나 받을 수 있고
  • Browser = HTTP를 통해 전달받은 정보를 사용자에게 보여줄 수 있다.

Browser 주소창에 넣는 문장이나 요청의 형식은 URL(Uniform Resource Locator)이라고 불리며 작성에 대한 약속된 기준이 있다.

URL의 작성 방식

  • scheme: 약속된 방식의 이름 (ftp, http …)
  • host: 접속하려는 페이지의 주소
  • port: 접속하거나 정보를 주고 받는데 쓰이는 통로의 구분
  • path: 해당 주소의 어떠한 파일로 통하는 경로
  • query: 추가로 요청할 때 무엇을 요청할지에 대한 정보

HTML

  • DOM을 구성하는 가장 기본적인 언어
  • 태그 안에 생성되어야하며, 태그와 태그 내에서 내용을 작성
  • 약속된 의미에 맞도록 태그를 사용해야 컴퓨터가 처리하기 좋음

DOCTYPE

마크업 언어에서 문서 형식을 정의하며, 웹 브라우저가 어떤 HTML 버전의 해석 방식으로 페이지를 이해하면 되는지 알려주는 용도

META

HTML문서의 제작자, 내용, 키워드, 화면 정보 등 여러 정보를 검색 엔진이나 브라우저에게 제공

## branch models

- **git flow (★) : 현재 강의에서는 주로 git flow를 다룰 것임**
- github flow
- gitlab flow

#Git Flow

## Git flow 작업 순서

- git flow init ⇒ git flow 초기화 셋팅
- git flow feature start {branch name} ⇒ 기능 branch를 생성 후 기능 개발 시작
- 개발이 끝나면 git flow feature finish {branch name}로 branch 제거
- git flow release start {release name} ⇒ 릴리즈 기능 시작
- git flow release finish {release name} ⇒ 릴리즈에 대한 확인 후 commit 메세지 작성, 해당 branch는 삭제되니 merge 메세지 작성
- 작업한 내용을 push를 안하면 github는 모르기때문에 git push -u origin {branch name} (여기서는 main branch가 아닌 작업했던 branch name을 기입)
- 이후 main branch로 가서 똑같이 git push origin main
- tag를 알리기 위해 git tag  를 입력 후 git push —tags로 버전도 같이 push
**ex) 첫 git flow tags link :** [https://github.com/dbstjddbwls/branch-practice/tags](https://github.com/dbstjddbwls/branch-practice/tags)

## Git Rename

- mv {file} {rename file}을 하고 git status를 하면 rename이 아니라 delete라고 인식한다.
- 이를 방지하고싶다면 git mv {file} {rename file}이라고 앞에 git을 넣어줘야함

## Git Restore

- 수정 후 commit까지 했는데 다시 되돌려야할때
- git restore {file name} 를 통해 바로 이전버전 커밋으로 되돌림

## Git Unstaging

- add를 취소하기
- add한 파일을 다시 unstaging하려면 git reset HEAD {file name}
    - 수정한 내용이 사라진건 아님

- restore와 reset의 차이점은 내용을 이전 버전으로 수정 하냐안하냐의 차이로 나뉘어진다
- 단 reset은 내 repo에서 삭제해도 다른 팀원의 repo에는 살아있을 수 있기 때문에 쓰지 않는걸 권장
    - reset의 대안은 **revert**를 사용

## Git Revert

- git revert —no-commit HEAD~{num}
- {num} 수 만큼 가장 최근 기준으로 이전에 커밋한 내용을 되돌림
- reset과의 다른 점은 revert의 경우 가장 최근의 커밋 내역까지 되돌리며 삭제까지 진행.
- 즉 파일 생성→ add→ status→ commit  순으로 보통 진행했다면 commit →status →add →파일 삭제로 되돌아감
- revert했다면 꼭 사유를 commit 메세지에 적을 것

- Git commit —amend

GIT

Git은 공식 사이트가 아닌 gitforwindow.org에서 다운받는게 여러 기능이 포함된 상태로 다운받을 수 있다

git bash 간단 사용법

  • cd : 경로이동을 할 때 사용
  • ls : 현재 경로에 있는 파일 목록이나 정보를 확인
  • mkdir : 현재 경로에 폴더를 생성
  • touch : 현재 경로에 파일을 생성
  • mv : 파일 이동에 사용. 파일의 이름 변경도 가능함
  • cp : 파일의 복사
  • rm : 파일의 제거

Vim

Git에 관련된 GUI를 사용 안한다면 git bash에선 vim을 사용하여 파일의 관리를 진행

Markdown systax

  • hashtag(#) : heading text
  • huphen (-) : unorder list
  • numbers(1.) : order list
  • plain text : paragraph

Git commit시 주의할 점

  1. git commit -m “Text” 사용금지
  2. git add . 사용금지
    => 모두 작업 시 이슈가 생기면 처리하기 힘들어진다

Git commit시 선호 규칙

  1. commit의 제목은 해당 commit을 설명하는 하나의 구나 절로 완성
  2. casebycase(X) => Case by Case(O)
  3. prefix
    • feat : 기능 개발 관련
    • fix : 오류 개선, 디버깅
    • docs : 문서화 파일 추가
    • test : test 내용
    • conf : 환경설정에 관한 내용
    • BREAKING CHANCE : 특정 일자부터 막히는 서비스에 대해
    • refactor : 기존 작동되는 작업을 특정 목적애 의해 재수정
    • build : build에 대한 안내

Hexo 사용해보기

  8. _config.yml에 접속하여 title, author, url, deploy… 수정

  10. npm install hexo-theme-next –sabe를 설치하여 next theme 활성화

  11. _config.yml에 theme를 수정

  12. hexo clean&&hexo generate를 실행

  13. hexo deploy로 url 및 repo github에 업로드
